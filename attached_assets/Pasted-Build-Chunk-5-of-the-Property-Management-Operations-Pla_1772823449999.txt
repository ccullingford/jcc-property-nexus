## Architecture requirements

### Backend
Add services such as:
- `contactService`
- `contactIdentityService`
- `contactSearchService`
- `contactTimelineService`

Responsibilities:

#### contactService
- create and update contacts
- retrieve contact detail
- manage linked phone numbers and email addresses

#### contactIdentityService
- normalize email addresses to a consistent lowercase format
- normalize phone numbers to E.164 where possible
- match contacts against message senders/recipients
- support safe manual linking/unlinking

#### contactSearchService
- search by name
- search by email
- search by phone number

#### contactTimelineService
- aggregate related tasks, notes, and email-thread activity for a contact
- return timeline data for the contact detail page

### API endpoints
Create endpoints such as:

- `GET /api/contacts`
- `POST /api/contacts`
- `GET /api/contacts/:id`
- `PATCH /api/contacts/:id`
- `GET /api/contacts/:id/timeline`
- `POST /api/contacts/:id/phones`
- `POST /api/contacts/:id/emails`
- `POST /api/threads/:id/link-contact`
- `POST /api/threads/:id/unlink-contact`

Support contact search via query params such as:
- name
- email
- phone

## Database requirements

Create migrations for the following tables.

### contacts
Fields:
- `id`
- `display_name`
- `contact_type`
- `primary_email` nullable
- `primary_phone` nullable
- `notes` nullable
- `created_at`
- `updated_at`

Allowed contact types for this chunk:
- `Owner`
- `Tenant`
- `Vendor`
- `Board`
- `Realtor`
- `Attorney`
- `Other`

### contact_emails
Fields:
- `id`
- `contact_id`
- `email`
- `is_primary`
- `created_at`

Requirements:
- store normalized lowercase emails
- enforce uniqueness where appropriate

### contact_phones
Fields:
- `id`
- `contact_id`
- `phone_number`
- `label` nullable
- `is_primary`
- `created_at`

Requirements:
- store normalized E.164 when possible
- support labels such as Mobile, Office, Dispatch

### thread_contacts
Fields:
- `id`
- `thread_id`
- `contact_id`
- `relationship_type` nullable
- `created_at`

This allows explicit linking between threads and contacts when auto-matching is imperfect.

Add proper foreign keys and indexes.

## Identity and normalization requirements

### Email normalization
- lowercase all stored emails
- trim whitespace
- use normalized values for lookup and matching

### Phone normalization
- normalize to E.164 where possible
- preserve raw input only if needed for display, but use normalized value for matching
- support multiple phone numbers per contact

### Matching behavior
The system should be able to:
- find an existing contact by sender email
- find an existing contact by recipient email
- later support phone matching cleanly
- allow manual correction if the match is wrong or missing

Do not over-automate matching in a risky way. Keep manual controls available.
