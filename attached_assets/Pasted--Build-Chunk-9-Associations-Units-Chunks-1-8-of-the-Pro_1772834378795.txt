# Build Chunk 9 — Associations & Units

Chunks 1–8 of the Property Management Operations Platform are already implemented.  
Do NOT rebuild earlier chunks unless required to support the features below.

Existing architecture:

Frontend: React  
Backend: Node.js  
Database: PostgreSQL  
Email: Microsoft Graph  
Authentication: Microsoft Entra ID

This chunk focuses on **property context**, specifically **Associations and Units**, so communications, contacts, and issues can be tied to the correct HOA and unit.

---

# Goals of Chunk 9

Add a dedicated property-context layer so the platform can support:

- associations
- units within associations
- linking contacts to associations and units
- linking issues to associations and units
- showing association/unit context in threads, contacts, and issues
- searching and filtering by association and unit

This chunk does NOT include:

- RingEX integration
- AI features

---

# Part 1 — Associations Data Model

Create a new `associations` model.

## Requirements

An association represents an HOA or community.

### associations
Fields:
- `id`
- `name`
- `code` nullable
- `mailbox_id` nullable
- `address_line_1` nullable
- `address_line_2` nullable
- `city` nullable
- `state` nullable
- `postal_code` nullable
- `notes` nullable
- `is_active` boolean default true
- `created_at`
- `updated_at`

## Purpose
This lets the system understand which HOA/community a conversation or issue belongs to.

Examples:
- Arlyn Circle Condominiums
- NoDa Landing
- Candlewood HOA

---

# Part 2 — Units Data Model

Create a new `units` model.

## Requirements

A unit belongs to a specific association.

### units
Fields:
- `id`
- `association_id`
- `unit_number`
- `building` nullable
- `street_address` nullable
- `notes` nullable
- `is_active` boolean default true
- `created_at`
- `updated_at`

## Constraints
- `unit_number` should be unique within an `association_id`
- add indexes for `association_id` and `unit_number`

## Purpose
This gives precise property context for residents, issues, and communication history.

---

# Part 3 — Contact Association and Unit Linking

Extend the contacts layer so a contact can be connected to an association and optionally a unit.

## Recommended first implementation
Keep this simple and practical.

### Add to contacts
Fields:
- `association_id` nullable
- `unit_id` nullable

This supports:
- one primary association per contact
- one primary unit per contact

That is sufficient for the initial operational workflow.

## Requirements
- contacts can exist without an association or unit
- if `unit_id` is set, it must belong to the same `association_id`
- validate this in backend service logic

## Future flexibility
Design cleanly so this can later expand to many-to-many relationships if needed, but do not overengineer it now.

---

# Part 4 — Issue Association and Unit Linking

Extend the issues layer so each issue can optionally be connected to an association and unit.

### Add to issues
Fields:
- `association_id` nullable
- `unit_id` nullable

## Requirements
- issue can exist without association or unit
- if `unit_id` is set, it must belong to the same `association_id`
- issue creation/edit UI must support selecting these fields

## Purpose
This allows issues like:
- Roof leak — Unit 204
- Parking dispute — Unit 17B
- Vendor invoice issue — Association-wide

# Part 13 — Data Integrity Rules

Enforce these rules in backend services:

1. A unit must belong to an association.
2. A contact’s unit must belong to the selected association.
3. An issue’s unit must belong to the selected association.
4. Deleting or deactivating an association must not silently corrupt linked data.
5. Deleting or deactivating a unit must be prevented or carefully handled if linked records exist.

Prefer soft handling / validation over destructive behavior.
