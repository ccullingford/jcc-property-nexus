## Architecture requirements

### Backend
Add services such as:
- `issueService`
- `issueLinkService`
- `issueTimelineService`
- `issueQueryService`

Responsibilities:

#### issueService
- create issues
- update issues
- retrieve issue details
- manage status and priority

#### issueLinkService
- link threads to issues
- unlink threads from issues if needed
- link tasks to issues
- unlink tasks from issues if needed

#### issueTimelineService
- aggregate issue-related activity from:
  - linked threads
  - linked tasks
  - notes
  - status changes
- return timeline data for the issue detail page

#### issueQueryService
- list issues
- filter by status
- filter by priority
- filter by assignee if applicable through linked tasks or future ownership

### API endpoints
Create endpoints such as:

- `GET /api/issues`
- `POST /api/issues`
- `GET /api/issues/:id`
- `PATCH /api/issues/:id`
- `POST /api/issues/:id/link-thread`
- `POST /api/issues/:id/unlink-thread`
- `POST /api/issues/:id/link-task`
- `POST /api/issues/:id/unlink-task`
- `GET /api/issues/:id/timeline`

Support filtering on issue lists such as:
- status
- priority
- open only
- closed only

## Database requirements

Create migrations for the following.

Fields:
- `id`
- `title`
- `description` nullable
- `contact_id` nullable
- `status`
- `priority`
- `created_by_user_id`
- `assigned_user_id` nullable
- `created_at`
- `updated_at`
- `closed_at` nullable

Allowed statuses for this chunk:
- `Open`
- `In Progress`
- `Waiting`
- `Resolved`
- `Closed`

Allowed priorities for this chunk:
- `Low`
- `Normal`
- `High`
- `Urgent`

### issue_threads
Fields:
- `id`
- `issue_id`
- `thread_id`
- `created_at`

### Optional issue_notes table

1. create a dedicated `issue_notes` table, or
2. extend the existing notes system cleanly so notes can belong to an issue as well as a thread

If creating `issue_notes`, fields:
- `id`
- `issue_id`
- `user_id`
- `body`
- `created_at`
- `updated_at`

Requirements:
- do not duplicate note logic unnecessarily
- preserve compatibility with existing thread notes

### Task linkage
If tasks already contain `issue_id`, use that field directly.
If not, add nullable `issue_id` to `tasks`.

Add proper foreign keys, indexes, and uniqueness constraints where appropriate.

## Workflow requirements

### Create issue from thread
From a thread, user should be able to:
- click Create Issue
- prefill title from thread subject if sensible
- optionally attach description
- optionally connect a linked contact if one exists
- set priority
- create the issue and link the current thread to it

### Manage issue
Users should be able to:
- update title and description
- update status
- update priority
- optionally assign issue owner if this field is included
- close or resolve issue

### Link related work
Users should be able to:
- link additional threads to an issue
- unlink threads when needed
- link tasks to an issue
- unlink tasks when needed

### Issue timeline
Issue detail page should show a timeline composed from real data such as:
- issue created
- issue status changed
- linked thread activity
- linked task activity
- issue notes

This timeline can be simple in this chunk, but it must be real and persisted.
