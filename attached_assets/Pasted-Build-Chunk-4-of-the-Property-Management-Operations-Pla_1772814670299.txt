## Architecture requirements

### Backend
Add services such as:
- `taskService`
- `taskAssignmentService`
- `taskQueryService`

Responsibilities:

#### taskService
- create tasks
- update tasks
- delete or soft-delete tasks if supported
- link tasks to threads

#### taskAssignmentService
- assign task to user
- reassign task
- track assignee changes if needed

#### taskQueryService
- list my tasks
- list team tasks
- list overdue tasks
- retrieve tasks related to a thread

### API endpoints
Create endpoints such as:

- `POST /api/tasks`
- `GET /api/tasks`
- `GET /api/tasks/:id`
- `PATCH /api/tasks/:id`
- `DELETE /api/tasks/:id`
- `GET /api/threads/:id/tasks`

Support query params or filtering for:
- assigned to me
- team tasks
- overdue
- status

## Database requirements

Create migration for `tasks`.

### tasks
Fields:
- `id`
- `issue_id` nullable
- `thread_id` nullable
- `assigned_user_id` nullable
- `created_by_user_id`
- `title`
- `description`
- `status`
- `priority`
- `due_date` nullable
- `created_at`
- `updated_at`

Allowed statuses for this chunk:
- `Open`
- `In Progress`
- `Completed`
- `Cancelled`

Allowed priorities for this chunk:
- `Low`
- `Normal`
- `High`
- `Urgent`

Add proper foreign keys and indexes.

If activity logging already exists, use it for key task actions such as:
- task created
- task assigned
- task status changed

## Workflow requirements

### Create task from thread
From a thread, user should be able to:
- click Create Task
- prefill task title from thread subject if sensible
- optionally add description
- assign user
- set due date
- set priority

The task must store the linked `thread_id`.

### Task management
Users should be able to:
- update task title/description
- assign or reassign task
- change status
- change priority
- set due date

### Thread integration
From thread view:
- show related tasks
- allow creating a task

From task view/list:
- show linked thread if present
- allow navigation back to thread
