## Architecture requirements

### Backend
Extend the Node backend with clear services/modules.

Add services such as:
- `graphSyncService`
- `threadService`
- `messageService`
- `mailboxReadService`

Responsibilities:

#### graphSyncService
- connect to Microsoft Graph using the authenticated Microsoft context
- read message/thread data from the configured shared mailbox
- map Graph data to internal models
- support incremental sync design, even if the first implementation is basic

#### threadService
- store and retrieve synced thread records
- expose thread list APIs
- support filtering by mailbox

#### messageService
- store and retrieve messages for a given thread
- expose thread message APIs

### API endpoints
Create or extend backend endpoints such as:

- `GET /api/mailboxes`
- `POST /api/mailboxes/:id/sync`
- `GET /api/threads`
- `GET /api/threads/:id`
- `GET /api/threads/:id/messages`

The backend should read from Postgres after sync, not directly from Graph for every page render.

## Microsoft Graph integration requirements
Support the configured shared mailbox as the primary mailbox source.

This chunk should:
- use Microsoft Graph to read mailbox content
- sync threads/messages from the shared mailbox configured in the database
- support the existing authenticated user context from Chunk 1
- be structured so future mailbox types can be added later

If thread-level retrieval is awkward with Graph, it is acceptable to sync by message/conversation grouping, but the app must present a thread-based inbox experience.

## Sync behavior
Implement a practical first-pass sync strategy.

Requirements:
- allow manual sync trigger
- persist synced data to Postgres
- avoid creating duplicates
- update existing rows when messages change
- track latest message timestamp per thread

It is acceptable in Chunk 2 to use a simple sync approach rather than a full production-grade delta sync, but the code structure should clearly support future improvement.
