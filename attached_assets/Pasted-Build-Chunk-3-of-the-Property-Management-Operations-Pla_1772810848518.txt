## Architecture requirements

### Backend
Extend the existing Node backend with services such as:
- `threadAssignmentService`
- `threadStatusService`
- `noteService`
- `activityLogService`

Responsibilities:

#### threadAssignmentService
- claim a thread for the current user
- assign a thread to another user
- unassign a thread if needed
- enforce clean rules for assignment state

#### threadStatusService
- update thread status
- validate allowed statuses
- log status changes

#### noteService
- create and retrieve internal notes for threads
- ensure notes are staff-only application data, not email data

#### activityLogService
- record thread-related events
- support timeline/history display in thread view

### API endpoints
Create or extend backend endpoints such as:

- `POST /api/threads/:id/claim`
- `POST /api/threads/:id/assign`
- `POST /api/threads/:id/unassign`
- `PATCH /api/threads/:id/status`
- `GET /api/threads/:id/notes`
- `POST /api/threads/:id/notes`
- `GET /api/threads/:id/activity`

## Workflow rules

### Claim thread
- current user can claim an unassigned thread
- claiming sets `assigned_user_id` to current user
- log activity

### Assign thread
- authorized user can assign thread to another user
- log activity

### Unassign thread
- authorized user can remove assignee
- log activity

### Status change
- user can change thread status
- log old and new status in activity metadata

### Internal notes
- notes belong to the thread
- notes are visible only inside the app
- notes are never synced to Outlook/Microsoft Graph as email content
